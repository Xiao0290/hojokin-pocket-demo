## 1. 产品是什么

**补助金ポケット** = 面向日本中小企业的「AI 补助金诊断 + 申请」移动 App。现状：React 可点击 demo（壳）已完成，正在规划本实装。

核心流程（12 画面）：
1. 输入公司 URL → 2. AI 解析公司业务 → 3. 与「2,847 制度」比对，给出可申请补助金一览（上限额/采择率/截止/剩余天数）→ 4. 补助金详情（补助上限/率/截止、申请步骤清单）→ 5. AI 自动生成事业计划书（章节化、~4,200字、流式）→ 6. 选择申请方式（自办 68% vs 委托专家 89% / +21pt）→ 7. 找专家（行政書士，AI推荐/采择实绩/着手金0円+成功报酬20%）→ 8. 依赖内容确认（采择时才收费）→ 9. 与专家聊天 → 10. 申请状况时间线 → 11. 提交 jGrants → 12. 提交完成（受付番号）。

底部 4 个 Tab：诊断 / 申请 / 消息 / 我的。

---

## 2. 产品定位（当前结论）

> 对外 = 「资金调达 App」，内部 = 「行政書士マーケットプレイス + 採択データ基盤」。

- **不要**做成「AI 替你写并提交」的 SaaS——会因行政書士法让平台自身成为违法主体（见第 4 节）。
- 角色三分（合规核心）：**AI = 本人的起草辅助工具** / **行政書士 = 资格者，书类的作成与提出主体** / **平台 = 撮合 + 本人作业辅助，向行政書士收费（非从用户处收书类对价）**。
- 真正的护城河（moat）：不是 2,847 制度 DB（国数据免费可复制），而是 ① 自治体长尾爬取 ② 要件 PDF → 机器可判定 JSON ③ 采择率独自收集 ④ 最终是自家「申请→采否」闭环数据。

## 3. 用户与市场（要点）

**需求侧（中小企业经营者）痛点**：约 75% 不知道自家可用的新设补助金；最大障碍是「申请流程复杂」+「事业计划书写不出」；gBizID 取得耗时（无 My Number Card 时邮寄审查 1~2 周、旺季 2~3 周）；不採択（落选）恐惧。两类人格：①叩き上げ社长（高龄、现场为主、手机仅 LINE）②成长型年轻经营者（数字化、决策快）。

**供给侧（行政書士/士业）痛点**：获客依赖广告/介绍、不稳定；初次咨询质量参差、低效。平台对他们的最大价值 = 提供「业务已整理、适配制度已锁定、本人已写到草稿」的**温热 lead**。

**竞争现实（已存在，差异化薄）**：
- 检索/诊断：ミラサポplus・J-Net21（公的免费）、补助金ポータル、补助金AI
- AI 计划书：补助金Express(旧 补助金.AI)、补助金フラッシュ、franca.ai 等
- 士业撮合：补助金ポータル、补助金幹事、补助金コネクト、ミツモア型
- 上游（最强）：freee / マネーフォワード（已握有客户决算数据）、Stayway「补助金クラウド」（与地银/自治体合作 B2B2C 批量获客）

→ 差异化只剩 4 点：(a) 移动端 × URL 一键诊断的低摩擦 (b) 采择率可视化 + 「自办 vs 委托」转化设计 (c) 着手金0円·成功报酬20%·不採択免费的透明撮合 (d) **最贴合法改正的合规设计**。

## 4. 已验证的关键约束（深度调研需在此基础上加深/更新）

| 主张 | 判定 | 置信 | 含义（对产品） |
|---|---|---|---|
| **改正行政書士法**（2026/1 施行）会触法 | ✅ confirmed | 高 | 报酬获得下「作成」事业计划书/申请书是行政書士独占业务，**「任何名目」**的对价（系统使用费/成功报酬/订阅）都算。罚则 1 年以下拘禁刑/100万円以下罚金 + 法人两罚。→「AI 自动生成并收费」「平台抽成功报酬」按原样实装 = 平台违法。**要再验证**：最新条文解释、总务省/日行连指引、グレーゾーン解消制度的具体回答。 |
| jGrants 公开 API 可自动提交 | ❌ refuted | 高 | 公开 API 仅只读 GET（检索/详情）。提交/状态/受付番号无 API；gBizID 民间系统连携被规约禁止。→ 提交无法自动化，只能引导用户手动 or 走官方「代理申請」功能（2025-01-30 上线）。**要再验证**：デジタル庁是否有开放提交 API / CSV 双向连携的路线图。 |
| 2,847 制度 DB 需分散爬取，是 moat | ✅ confirmed | 高 | jGrants 公募中仅约 147 件、以国补助金为主；自治体「逐次移行」中、1,700+ 未全收录。真母数远大（竞品年收约 3 万件）。国数据免费可复制 ≠ moat，moat 在运营蓄积。「2,847」具体出处**要再验证**。 |
| 采择率 68%/89%/+21pt 有景表法风险 | ⚠️ 部分 | 中 | 景表法本质针对「一般消费者」，纯 B2B 未必适用；但面向个人经营者的 B2C 移动 App 偏「适用侧」。无合理根据的断定 = 优良误认风险。**要再验证**：B2B/个人事业主场景的适用边界、不实证广告规制实务。 |
| 成功报酬 20% 抽成会挤压士业、致场外交易 | ⚠️ 部分 | 高 | 相场：成功报酬 10~20%（中位 15%）、着手金 5~30万（中位 15万）；完全成果报酬型通常 15~25%。平台再从中抽 % → 士业实得低于标准，且诱发直接交易。另：对行政書士的介绍费/手续费受职业规程约束（非弁/周旋疑义）。 |

## 5. 技术栈（按功能域，结论）

| 域 | 推荐 | 关键点 |
|---|---|---|
| 前端·移动 | **Expo / React Native**（需求侧）+ **Next.js**（士业 PC 管理）+ Turborepo 单仓 + TS strict | 现壳仅「设计」可复用，代码近乎重写 |
| 后端 | **NestJS(TS)** on AWS东京，REST + SSE | 全 TS 统一；模块化表达多侧权限 |
| 数据 | **Aurora Serverless v2 (PostgreSQL + pgvector)** + OpenSearch + S3 | 单库放制度/向量/事务 |
| LLM | **Amazon Bedrock Claude（ap-northeast-1, JP Geo）** | 机微数据不出境 |
| AI 诊断 | Playwright 抓取（仅本人自社 URL）→ trafilatura → Claude 结构化抽取 → **混合匹配**（pgvector 粗筛 → 规则硬判定 → LLM 仅解说） | 硬要件决定式判定，杜绝不适格混入 |
| 数据基盘 | jGrants 公开 API 日次 ETL + 厚労省 + 自治体分阶段 + **PDF→要件 JSON（LLM+人审）** | program⇄round 1对多；provenance 必填 |
| AI 计划书 | Claude Opus 流式 + Structured Outputs + 制度模板 + RAG(采择事例) + eval 护栏 | 数值绑定来源、未据则降「要确认」 |
| 撮合 | 资格验证(行政書士登录番号无公开 API→人审)、eKYC(JPKI/TRUSTDOCK)、Stripe Connect(不滞留资金)、状态机(不採択无法发起收费) | 法令准拠架构先于一切 |
| 实时 | Stream Chat + 自社 DB 镜像(审计/哈希链) | 守秘义务·证据保全 |
| 提交 | 只读 API + **「提交 kit」生成** + gBizID 仅引导 + 映射官方代理申請 | 提交不可自动化 |

## 7. 商业模式假设（待验证）

- 主收益柱 = **需求侧/供给侧 SaaS**（不依赖采否、现金流早、合规高）。
- 撮合课金 = **挂载费/月费/lead 费**，而非抽成功报酬 %（规避周旋对价 + 手取挤压）。
- 「着手金0円+成功报酬20%」作为士业向用户的报价展示，平台不抽该 20%。

## 8. ★ 待深挖的研究问题（交给深度调研）

> 每条都尽量「可被一手资料证实/证伪」。建议优先 A、B、C。

### A. 法规（最高优先 — 决定事业结构的生死）
1. 改正行政書士法（令和7年法律第65号，2026/1/1 施行）的**条文原文与官方解释**：报酬「いかなる名目」具体到什么程度？「テンプレート提供」「大量情报提供」是否真被视为违法？一手来源（总务省、日本行政書士会連合会、官报）。
2. **グレーゾーン解消制度**里关于补助金申请支援/事业计划书作成的产业省·总务省**回答原文**：合法的「助言/添削/信息提供」与违法的「作成」边界线在哪？
3. 合法可成立的**事业 scheme**：平台向行政書士收「系统利用料/送客费」、用户与行政書士直接委任——这种结构有无判例/官方见解背书？是否需要平台自身持有行政書士法人？
4. 弁護士法 72 条（非弁）与本模型（AI 生成 + 撮合 + 收费）的关系。
5. IT导入补助金的「IT导入支援事业者」登录制——平台自身注册成支援事业者，是否是一条合法捷径？最新（2026 改称「デジタル化・AI導入補助金」）的支援事业者制度细则。

### B. jGrants / gBizID 的连携路线图
6. デジタル庁是否公布过**提交（POST）API / CSV 双向导入**的开放计划与时间表？（FY2025 以降 roadmap 原文）
7. gBizID 对**民间系统**的 OIDC 连携：现状禁止，是否有放开计划？RP 登录的实际门槛（资格、周期、成本）。
8. 官方「代理申請」功能的**完整规格**：委任 1:1 限制、哪些补助金支持、本人最终提交按钮是否必须、士业能完成到哪一步。
9. gBizID プライム 取得的最新流程与耗时（My Number Card 即日 vs 邮寄），以及普及率/离脱率数据。

### C. 补助金数据源与许可（moat 的可行性）
10. 各一手数据源的**收录范围与再配布/商用许可**：jGrants 公开 API、ミラサポplus、各自治体开放数据（IMI 共通词汇/推荐数据集）、厚労省 esop。
11. 「日本补助金/助成金的真实母数」与分布：国 / 1,700+ 自治体 / 财团 各自规模；竞品（スマート補助金、ミライ サポート等）声称的收录量与更新频率，可信度。
12. 采择率数据的**公开状况**：哪些制度/回次公开采择率与采择者一览（出处），可机读程度。

### D. 竞争对手深挖
13. Stayway「补助金クラウド」、freee/マネーフォワード 补助金功能、补助金Express、补助金ポータル/コネクト 的**最新功能、定价、获客渠道、合规处理**（尤其法改正后如何调整）。各自的弱点与本产品可乘之机。
14. 是否已有「移动端 × URL 一键诊断 × 透明士业撮合」的直接对标者？

### E. 成功报酬 / 介绍费的合法结构
15. 行政書士会的**报酬规程/广告规程**对成功报酬、介绍费、平台分成的具体约束。
16. 类似两侧平台（士业撮合）如何设计课金以规避「周旋对价」认定（ミツモア、ココナラ等的实做与规约）。

### F. 采择率与景表法
17. 景表法对 **B2B / 个人事业主**场景的适用边界、不实证广告规制（法 7 条 2 项）实务、消费者庁近年对「No.1/效果/实绩」表示的执法案例。
18. 「+21pt 上がる」这类**因果断定**表达的合规改写与举证要求。

### G. 市场规模 / 单位经济
19. 日本补助金申请支援市场规模、行政書士/认定经营革新等支援机关数量、目标 SMB 数量（TAM/SAM/SOM 拆解）。
20. 各主要制度的平均采择额、采择率近年趋势（ものづくり/持続化/事業再構築/IT導入/省力化），用于 GMV 与单位经济测算。

### H. 邻接扩张
21. 「资金调达 OS」扩张到**融资斡旋**的金融规制（贷金业法、与日本政策金融公库/信用保証協会/民间银行的合作模式）。
22. 税务顾问、许认可、助成金（社労士管辖）等邻接领域的合规与撮合可行性。
